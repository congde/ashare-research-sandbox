# A 股个人研究工具：用户与竞品调研

> 虚构教学标的「示例科技（600001）」仅用于课程演示。本报告支撑
> [product-brief.md](product-brief.md) 与 [prd.md](prd.md) 的范围决定。

## Facts

- F1: 东方财富、同花顺等主流 App 提供 A 股实时行情、资讯、社区讨论与部分
  条件单能力，面向已开户的个人投资者。Source: 各平台公开产品介绍页（2026-03
  抽样，未登录试用）。
- F2: Wind、Choice 等专业终端提供更深的数据与研报，但通常面向机构或高成本
  订阅用户，个人学习者难以在无预算情况下复现完整工作流。Source: 公开定价与
  产品说明页摘要。
- F3: 课程目标用户（非程序员学习者）在调研访谈中的共同表述是：「我能看到
  很多观点，但分不清哪些有来源、历史策略结果意味着什么、以及为什么不能
  直接照着做。」Source: 课程内部用户访谈纪要（3 人，2026-03，非代表性样本）。
- F4: 参考项目 [web3-trading](https://github.com/congde/web3-trading) 采用
  FastAPI、外部行情 API、MongoDB 与 Agent 编排，适合有运维与密钥管理能力的
  团队；其 `example/04_backtest_demo.py` 展示了「录制因子快照 → 回测 →
  导出报告」的完整链路，但依赖 API Key 与数据库。Source: 仓库 README 与
  example 脚本（2026-06 克隆阅读）。
- F5: 监管与合规语境下，面向不特定公众提供具体买卖建议或代客下单，需要相应
  资质；教学项目必须明确「研究 + 模拟」边界，不得暗示收益保证。Source: 课程
  合规审查备忘（内部）。

## Inferences

- I1: 目标用户的第一痛点不是「缺少更多行情」，而是**缺少可复查的研究与模拟
  流程**——能把来源、解释、历史样本结果和风险说明放在同一页面。Supports: F1,
  F3
- I2: 第一版若复制 web3-trading 的全栈形态，会把大量精力耗在密钥、数据库与
  部署上，偏离「证明 Codex 交付流程」的课程目标；应保留其「研究摘要 + 回测
  报告 + 边界提示」的产品形状，但用固定样本与标准库实现。Supports: F4, F5
- I3: 竞品 App 的「一键跟单」「智能荐股」能力与课程目标冲突；第一版应明确
  **不做**实时荐股与账户连接，以差异化定位「学习者验证台」而非「交易助手」。
  Supports: F1, F5

## Recommendations

- R1: **Go（继续）**——做「A 股投资研究与模拟策略验证台」第一版：单只虚构
  标的、固定数据集、带来源研究摘要、双均线回测、浏览器界面与自动验收。Supports:
  I1, I2
- R2: 实施计划采用三层架构（数据包 → 引擎 → 界面/API），每 milestone 带证据门；
  详见 [plan.md](plan.md)。Supports: I2
- R3: 用户测试聚焦「能否区分历史样本结果与未来收益」，不以策略是否盈利作为
  通过条件。Supports: I3, F5

## Unknowns

- U1: 真实个人投资者是否愿意使用「无实时行情」的静态样本工具进行学习。
  Impact: 可能影响产品形态向「可导入自选 CSV」演进。Next check: 完成
  [user-test.md](user-test.md) 后复盘。
- U2: 固定 35 日样本是否足以让学习者理解回撤与交易次数，而不产生过度外推。
  Impact: 可能需要增加第二段样本或对比基准说明。Next check: 第 14 讲用户测试
  记录。
- U3: 未来若接入真实公告 API，来源卡与合规审查成本将显著上升。Impact: 可能
  推迟到 Playbook 中的「明确拒绝」项。Next check: 第 20 讲 Playbook 边界表。

## 竞品对照（摘要）

| 产品类型 | 代表 | 优势 | 与本项目关系 |
|---|---|---|---|
| 零售 App | 同花顺、东方财富 | 实时行情、社区、交易闭环 | 第一版**不做**账户与荐股 |
| 专业终端 | Wind、Choice | 深度数据与研报 | 成本与复杂度超出 MVP |
| Agent 量化参考 | web3-trading | 因子回测、报告导出、可扩展 | 借鉴**报告形状**与回测分离，不复制依赖栈 |
