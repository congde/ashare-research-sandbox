## 二、为什么要做这门课

大多数 AI 工具教程仍然把 Codex 讲成「更聪明的 Copilot」：会补全、会改代码、会聊天。用户学会的是提示词技巧，而不是 **委托一个能独立推进、又能被验收的助手**。

实际使用中，人们普遍卡在四类问题：

1. **只会问，不会委托**：把 Codex 当搜索引擎或聊天框，任务边界、完成标准和验收方式从未写清。
2. **只会生成，不会交付**：助手给出了看起来合理的文字或代码，但没有来源、没有证据、没有可接手格式，结果无法进入下一步。
3. **只会写代码，不会当助手**：调研、整理、计划、数据处理、浏览器操作等日常任务从未进入 Codex 工作流；一旦不涉及代码，就回到手动模式。
4. **个人偶尔成功，无法复用**：缺少项目说明、Skill、Automation 和评测机制，成功经验不能稳定复制，更谈不上团队推广。

本专栏不按 Codex 功能菜单讲解，而是围绕一条通用委托链路组织训练：

> 接到任务 → 写 Brief → 选入口与上下文 → 委托 Codex → 审查产物 → 留下验收证据 → 必要时 Handoff → 沉淀为可复用工作流

课程要回答的核心问题不是「Codex 能不能写代码」，而是：

> **怎样把 Codex 当成个人助手，持续交付一份你自己敢用、别人敢接、事后能复查的结果。**

写代码、改 Bug、提 PR，是这条链路上的 **高难度任务类型**，不是 Codex 的唯一用途。

## 三、专栏与图书的关系

### 1. 图书：建立系统认知

图书围绕 AI 编程智能体的认知迁移、上下文工程、能力边界和团队治理展开，回答「为什么 Codex 不只是代码生成工具」。

### 2. 专栏：练成工作方式

专栏不沿用图书章节顺序，而是让读者在 20 讲里反复完成 **委托—验收—沉淀** 闭环。图书讲判断，专栏练动作。

| 对比维度 | 图书 | 专栏 |
|---|---|---|
| 核心定位 | 方法论与边界判断 | 个人助手式委托与验收 |
| 组织方式 | 按知识体系展开 | 按任务复杂度递进 |
| 主要载体 | 论述与案例框架 | Brief、Lab、Skill、验收脚本 |
| 代码角色 | 重要场景之一 | 进阶专篇，不是开篇默认 |

**一句话**：图书帮你理解 Codex 能做什么、不能做什么；专栏帮你把 Codex 练成日常助手。

## 四、目标读者

### 主受众（极客时间核心付费人群）

- 已用过 ChatGPT、Copilot、Cursor、Codex，但主要停留在「问答 + 局部生成」的 **开发者**；
- 希望 Codex 不只写代码，也能处理调研、写作、计划、数据整理等日常事务；
- 需要 Codex 进入 **代码库、Bug 修复、PR、Review、CI** 与团队协作流程；
- 想把个人助手经验沉淀为 Skill、Automation 和团队 Playbook 的 Tech Lead。

### 延伸受众

- 非开发岗位的知识工作者（产品、运营、研究、管理），可只学第一至三篇与第六篇，跳过代码专篇仍算主干结业。

### 前置基础

- 能使用电脑完成日常办公与简单文件操作；
- 开发者读者需具备基本 Git 与终端经验；
- 不要求有 Agent、MCP 或自动化经验。

### 三条学习路径（借鉴极客时间工程化专栏的灵活路径设计）

| 路径 | 适合谁 | 建议讲次 | 结业标准 |
|---|---|---|---|
| **助手路径** | 非开发 / 先练委托 | 1–10、18–20 | Brief、调研、写作、Skill、Eval、Playbook |
| **开发者路径** | 日常写代码的研发 | 1–17、19–20 | 在助手路径基础上完成热修复 / PR / CI |
| **负责人路径** | Tech Lead | 全文 + 重点 9–10、16–17、20 | 团队 Playbook、Automation、Eval |

### 学完后的能力

1. 把模糊需求整理成 Codex 可执行、可验收的 Brief；
2. 为不同任务选择合适的入口（App / CLI / Cloud）和上下文包；
3. 用 Codex 完成调研、写作、计划、数据整理、浏览器操作等 **非代码日常任务**；
4. 建立来源、结构、复算、截图等多类型验收习惯；
5. 当任务涉及代码库时，完成勘察、修复、功能开发与长任务交付；
6. 让结果进入 PR、Review、CI 和 Handoff 等团队协作流程；
7. 将稳定流程沉淀为 Skill、Automation，并用 Rubric / Eval 持续改进。

## 五、课程设计原则

### 1. 助手优先，代码后置

前 10 讲建立 **个人助手** 的委托与验收肌肉；代码库任务从第 11 讲起作为「助手能力进入工程场景」的延伸，而非课程默认起点。

### 2. 按任务类型组织，不按产品功能组织

App、CLI、Cloud、MCP、Skills、Automations 等能力，只在当前任务需要时出现。

### 3. 每讲一个可验收交付物

每讲结束时，读者应留下 **可被第三方检查** 的产物：Brief、报告、计划、数据表、Handoff 文档、Diff、PR 或 Eval 记录——而不是一段聊天记录。

### 4. 解释决策，不是命令流水账

正文说明：为什么这样委托、给了什么上下文、验收标准是什么、哪些判断不能交给 Agent。

### 5. 配套资产可运行、可检查

- 助手类 Lab：结构校验、来源检查、Rubric 评分；
- 工程类 Lab：`make lab-xx` 确定性验收；
- 两类 Lab 共用同一套 Brief 模板与 Handoff 格式。

### 6. 一条任务线贯穿前四讲（借鉴 GitHubSentinel 式项目主线）

第 1–4 讲围绕 **同一份「笔记工具调研」任务** 推进：写 Brief → 选入口与工作区说明 → 委托调研 → 验收报告。避免前讲只讲概念、后讲才做闭环的断裂感。

### 7. 对齐 Codex 官方工作循环

课程实操对齐 OpenAI 推荐的 **Inspect → Plan → Edit → Verify → Report** 循环（见 [Working with Codex](https://openai.com/academy/working-with-codex/)）：Brief 对应 Plan，验收对应 Verify，Handoff / PR 描述对应 Report。

## 六、课程结构

> 结构参考：[Claude Code 工程化实战](https://news.qq.com/rain/a/20260131A02R6I00) 的「真实卡点驱动 + 多学习路径」；[GitHubSentinel Agent 实战](https://time.geekbang.org/course/intro/101061913) 的「同一项目逐章交付」；OpenAI Codex 的 **Inspect → Plan → Edit → Verify → Report** 循环。

### 开篇词｜别再把 Codex 当聊天框

说明「问答」与「委托」的区别，以及为什么个人助手也需要 Brief、验收和 Handoff——这些不是工程团队的专属概念。

---

### 第一篇｜委托基本功：先学会把任务交出去

第 1–3 讲建立委托语言；**完整非代码闭环在第 4 讲完成**（与第 1 讲同一调研任务衔接）。

**第 1 讲｜第一个委托：从「帮我想想」到 Brief**

以「笔记工具调研」为贯穿任务，讲清 Goal、Context、Constraints、Done when 与开放问题。配套 `labs/00-assistant-brief/`，用结构脚本验收 Brief 是否可委托。

**第 2 讲｜验收思维：怎么知道助手真的做完了**

在正式委托前定义验收形式——结构、来源、样本复算、命令输出——并完成一次人工放行或拒绝判断。先用 Brief 做「可委托性验收」，为第 4 讲的报告验收打底。

**第 3 讲｜入口、上下文与工作区说明**

按任务时长、交互频率、环境依赖选择 App、CLI 或 Cloud，并组织 `@` 附件与线程上下文。引入 **轻量版工作区说明**（个人文件夹或课程仓库的 `AGENTS.md` 思路）：Codex 每次开工前读什么、输出放哪里。交付 `entry-decision.md` 与工作区说明草稿。

**本篇交付结果**：一份可执行 Brief、验收清单、入口决策与工作区说明草稿（**不是**完整调研报告；完整闭环在第 4 讲）。

---

### 第二篇｜日常助手：调研、写作与计划

承接第一篇同一任务，完成第一次 **调研闭环**；再扩展写作与计划场景。

**第 4 讲｜调研与信息整理：多源材料 → 有来源的结论**

延续「笔记工具调研」任务：委托 Codex 执行 Brief，将结果分为事实、推断、建议、未知，并为事实绑定来源。配套 `labs/04-research/`。**本篇在此完成第一条非代码任务闭环。**

**第 5 讲｜写作与汇报：从草稿到可发版**

用结构约束处理周报、方案、邮件、汇报，明确哪些段落可委托、哪些必须人工润色，交付一份结构合规、可对外使用的写作产物。

**第 6 讲｜计划与拆解：模糊目标 → 可执行清单**

把模糊目标拆成里程碑、依赖与开放问题，输出可执行计划与 Handoff 草稿，让下一位接手者无需重读整段对话即可继续。

**本篇交付结果**：带来源的调研报告（闭环）+ 写作产物 + 可执行计划各一份。

---

### 第三篇｜连接世界：MCP、浏览器与自动化

让助手访问真实资料、操作真实界面，并在无人值守时按护栏推进任务。

**第 7 讲｜MCP：连接日历、网页、数据库与内部工具**

划分必需、按需、禁止三类上下文，通过 MCP 获取可追溯的外部信息。交付一次 MCP 委托记录与来源清单。

**第 8 讲｜浏览器与 GUI：可见操作也要留证据**

浏览器验收记录完整用户路径——起始状态、操作步骤、预期与实际结果、异常态——而不是只留一张结果截图。

**第 9 讲｜Skills：把「每周做一次」变成一键工作流**

从 **个人助手重复任务**（如周报、调研 Brief）中提炼 Skill：适用场景、标准步骤、模板与验收。配套 **`skills/weekly-brief/`** 与 `make lab-09`（助手向 Skill，不是代码库勘察）。

**第 10 讲｜Automations：人不在线时助手怎么推进**

为已经稳定的工作流设计触发条件、权限边界、审批 gate 与失败降级策略，交付一条低风险的 Automation 设计方案。

**本篇交付结果**：一个助手向 Skill、一条 Automation 设计、一份浏览器或 MCP 任务的完整证据链。

---

### 第四篇｜动手任务：终端、数据与代码库

当任务需要改文件、跑命令、读代码库时，沿用同一委托范式，追加工程化验收。

**第 11 讲｜终端与文件：数据处理与批量整理**

处理 CSV 汇总、批量重命名、日志提取等脚本化任务，讲清委托边界、抽样复算与异常记录，交付数据处理记录与样本复算证据。

**第 12 讲｜工程护栏：Rules、Hooks 与 Permissions**

在第 3 讲工作区说明基础上，把「不要乱改、不要泄露、不要绕过检查」变成可执行的 Rules、Hooks、Sandbox 与审批。配套根目录 `AGENTS.md` 与拦截案例。

**第 13 讲｜陌生代码库勘察：先交地图，再交代码**

在修改前先输出职责地图、调用链、测试覆盖与影响面分析，交付 `IMPACT.md`。配套 **`skills/repo-readiness/`** 与 `make lab-16`（工程向 Skill，放在进代码库前）。

**第 14 讲｜Bug 与热修复：最小变更 + 测试证据**

从复现、委托、**审 Diff** 到先红后绿与全量回归，完成最小热修复并控制修改范围。配套 `labs/01-first-ticket/` 与验收命令输出。

**本篇交付结果**：一份勘察报告或热修复交付。非开发者可将第 13–14 讲替换为「代码库只读分析 + 发现改写为工程 Brief」。

---

### 第五篇｜工程交付：当助手结果要进团队流程

服务开发者与 Tech Lead，将个人助手范式延伸到 PR、Review、CI 与并行协作。

**第 15 讲｜功能开发与长任务：竖切、Checkpoint 与回滚**

从用户行为切出跨文件竖切，并为重构、迁移等长任务建立行为基线、阶段验收点与回滚路径。（本讲合并两个相关场景，正文以一个竖切案例为主、长任务 checkpoint 为辅。）

**第 16 讲｜PR、Review 与拒绝复盘**

将背景、范围、测试证据、风险与未决事项组织为 PR 交付接口；让 Codex 参与 Review 并形成 findings 改评闭环；当 PR 被拒时，区分目标错位、证据不足、维护成本与流程约束，**重写 Brief 或拆分方案**（保留旧版高价值「拒绝复盘」专题）。

**第 17 讲｜CI 与并行：Worktree、Cloud、失败自修复**

用 Worktree 与 Cloud 隔离并行任务；CI 失败时自动分析但不自动放行，保持权限边界并交付 Handoff 文档。（本讲偏综合，以一条 CI 失败恢复案例串联并行与交接。）

**本篇交付结果**：一条进入团队质量流程的完整交付链（PR、Review、CI、Handoff 至少覆盖三项）。

---

### 第六篇｜沉淀与毕业：从个人助手到可持续工作流

证明工作流稳定有效，并导出可迁移的个人或团队 Playbook。

**第 18 讲｜Traces、Rubric 与 Eval：怎样证明助手真的变可靠了**

保存执行轨迹，设计评分标准，用多个样本比较工作流修改前后的表现。Eval 样本须覆盖 **至少一个非代码任务**；开发者额外覆盖一个代码任务。

**第 19 讲｜混合任务毕业练：调研 → 实现 → 文档 → 交接**

**双轨过关（同一 Brief 模板，不同交付物）：**

| 轨道 | 过关任务 |
|---|---|
| 助手轨 | 调研 + 写作/计划 + 文档 + Handoff（无代码 Diff） |
| 开发者轨 | 调研 + 热修复或小功能 + 文档 + Handoff + 验收命令 |

**第 20 讲｜个人 Playbook 与团队推广**

区分什么该沉淀为 Skill、什么该推广给团队、什么该明确拒绝；导出个人或团队 Playbook，并验证另一位读者仅依据 Handoff 文档即可继续任务。

**本篇交付结果**：Eval 报告、毕业交付包、Playbook。

---

### 结束语｜真正的过关，是下一件杂事也知道怎么委托

回顾六篇路径，给出「任务类型 → 入口 → Brief 要点 → 验收形式」决策卡；强调 Codex 的价值不在于多生成一些文字或代码，而在于 **你少收几次烂尾结果**。

## 七、单讲内容形态

| 模块 | 作用 |
|---|---|
| 真实任务 | 来自办公、调研、数据或研发的真实压力，而非虚构功能演示 |
| 委托示范 | 完整 Brief + 上下文选择 + 入口理由 |
| Codex 机制 | 只讲本讲任务真正用到的能力 |
| 验收演示 | 展示如何通过 / 如何拒绝助手结果 |
| 人工判断点 | 明确不能交给 Agent 的责任 |
| 翻车与恢复 | 常见失败（幻觉、越界、凑格式、测试凑绿）及恢复 |
| 过关任务 | 读者独立产出可检查交付物 |

## 八、配套仓库设计

`codexDemo` 从「纯研发训练场」扩展为 **助手 + 工程双轨训练场**：

| 类型 | 目录示例 | 验收方式 |
|---|---|---|
| 助手 Lab | `labs/00-assistant-brief/`、`labs/04-research/`、`labs/09-weekly-brief-skill/` | 结构脚本、来源检查、Rubric |
| 工程 Lab | `labs/01-first-ticket/`、`labs/16-repo-readiness-skill/` | `make lab-xx` |
| 共用资产 | Brief 模板、Handoff 模板、`AGENTS.md`、Skills | `make check` 统一入口 |

原则：

- 正文提到的文件、命令与验收必须与仓库一致；
- 非开发者路径不要求本地跑测试，但要求完成助手类 Lab 的文档验收；
- 开发者路径在第四篇起接入现有工程 Lab，不必重复建设。

## 九、差异化价值

与常见 AI 编程课 / 提示词课相比：

1. **定位差异**：Codex 是 **开发者的可验收个人助手**，不是更好的代码补全；代码是助手任务谱系中的一段，不是课程唯一主线。
2. **结构差异**：前 10 讲完成非代码助手闭环，后 10 讲按需进入工程深水区；第 1–4 讲用 **同一任务线** 完成第一次闭环，避免「讲概念 ≠ 做闭环」。
3. **方法差异**：统一 Brief → 委托 → 验收 → Handoff 范式，对齐 OpenAI Codex 的 Inspect / Plan / Verify / Report 循环。
4. **路径差异**：提供助手轨 / 开发者轨 / 负责人轨三条学习路径（参考 [Claude Code 工程化实战](https://news.qq.com/rain/a/20260131A02R6I00) 的灵活选讲设计）。
5. **证据差异**：每个重要方法都有可检查交付物；助手任务验收来源与结构，工程任务验收命令与 CI。
6. **边界差异**：明确哪些决策必须人做——合规、架构拍板、人际协调——助手只能列选项，不能替签。

## 十、预期成果

读者结业时应拥有：

- 一套跨场景复用的 **Brief 模板** 与 **验收清单**；
- 三类日常助手产物范例（调研、写作、计划）及证据链；
- 一个经过验证的 **Skill** 和一条 **Automation** 设计；
- 开发者额外拥有：`AGENTS.md`、勘察 / 热修复 / PR 交付经验；
- 一份 **Eval** 与 **个人 Playbook**（可选团队版）。

最终希望读者形成的习惯：

> **接到任何事，第一反应不是打开聊天框，而是写 Brief、选入口、定验收——无论这件事要不要写代码。**

## 附录：与旧版大纲的主要变化

| 维度 | 旧版（智能交付实战课） | 新版（个人助手实战课） |
|---|---|---|
| 默认身份 | 研发交付 Agent | 开发者的可验收个人助手 |
| 第一课 | 热修复代码工单 | 非代码 Brief（同一任务线） |
| 第一次闭环 | 第 1 讲 | **第 4 讲**（调研报告） |
| AGENTS.md | 第 4 讲（工程） | **第 3 讲**轻量版 + 第 12 讲工程深化 |
| 助手 Skill | 无 / 第 18 讲附录 | **第 9 讲** `weekly-brief` |
| 工程 Skill | 第 16 讲 | **第 13 讲** `repo-readiness` |
| PR 拒绝复盘 | 独立一讲 | 并入 **第 16 讲** |
| 非代码位置 | 第 18 讲附录 | 第 1–10 讲主线 |
| 毕业路径 | 单一 | **助手轨 / 开发者轨** 双轨 |

旧版工程 Lab 与 `docs/v2/` 正文可复用于第四、五篇，Brief 模板与 Handoff 格式向前延伸到第一篇统一复用。
