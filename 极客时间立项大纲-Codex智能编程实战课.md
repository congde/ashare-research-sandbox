**副标题**：写代码只是一种能力——把 Codex 讲透、用到工作流里  
**出品方**：袁从德  
**配套图书**：《Codex：AI 驱动的智能编程时代》（邮电出版社，见 `book/Codex.docx`）

## 一、课程概述

本专栏围绕 **26 章目录**展开，不以凑章节数为目标，而以**讲透**为目标：每章只解决一个问题——讲清一个 Codex 机制，或跑通一条完整工作流，学完能独立复现。

Codex 是面向工作的 Agent 伙伴。写代码只是其能力之一；它还能理解系统、审阅变更、连接外部工具、定时巡检、操作浏览器与电脑、跨天续作（[Codex for (almost) everything](https://openai.com/index/codex-for-almost-everything/)）。很多人装好后只会问几句、改几行，根本原因是**没摸清机制，只扫过菜单**。

**配套图书**回答「为什么」；**本专栏**回答「怎么调通」——在 [codexDemo](https://github.com/congde/codexDemo.git) 与真实工作流中，按目录逐章过关。

## 二、目标学员

想**真正用好** Codex 的开发者与技术负责人——遇到问题知道调哪项配置、开哪个入口、跑哪条验收，而不是碰运气提问。

## 三、课程目录

开篇词｜章节数量不是目的——每一章讲透一个机制或一条闭环，能复现才算过关

一、认知篇——先建立正确预期
> 过关标志：能画出 Codex 能力四象限，说清委托闭环四步

01｜开篇导读：书课分工、专栏过关标准、codexDemo 怎么用
02｜Codex 能力地图：四象限全景，写代码只占其一
03｜委托闭环深讲：上下文 → 任务 → 验收，感知 / 推理 / 执行 / 反馈

二、工作台篇——讲透 App 这条主战场
> 过关标志：能在 App 内完成一次带 Diff 审阅的完整委托

04｜App 工作台：线程、Diff 逐行审阅、Plans、分块 stage/revert
05｜执行环境一体讲：Local / Worktree / Cloud、Handoff、多线程并行
06｜全入口协同：CLI 脚本化、IDE 插件、Cloud 环境、移动端各何时上场

三、安全篇——讲透 Sandbox 与审批
> 过关标志：能写出一份适合自己项目的 `config.toml` 安全基线

07｜Sandbox 实战：`read-only` / `workspace-write` / 可写目录 / 网络边界
08｜审批体系：`approval_policy`、`execpolicy`、Auto-review 与 `config.toml`
09｜禁区与分级信任：五类绝不委托场景，按风险决定放权粒度

四、上下文篇——讲透 Harness 全栈
> 过关标志：在 codexDemo 跑通 Harness 最小链路

10｜AGENTS.md 与任务 Brief：让 Agent 看见什么、验收什么
11｜Rules、Hook、Memory：约束偏差、记住纠错
12｜Skills 深讲：编写、调用、与 Automations 组合

五、连接篇——讲透 MCP 与外部工具链
> 过关标志：能从 JIRA / GitHub 拉上下文完成一次跨工具委托

13｜MCP 与插件：GitHub、JIRA、Notion、Slack 上下文怎么拉齐
14｜模型选型与成本：推理深度、速度、账单，按任务匹配模型

六、工程篇——讲透代码库上的全链路
> 过关标志：独立完成「勘察 → 开发 → Review → CI」一条链

15｜理解仓库与功能开发：勘察地图 → 需求 → PR
16｜Bug、测试与重构迁移：最小修改、回归、大改动拆任务
17｜PR 双向闭环：输出 Review findings + 处理 GitHub Review 意见
18｜CI 自修复：`codex exec`、GitHub Action、流水线集成

七、自动化篇——讲透后台协作
> 过关标志：配置一条 Automation 并完成收件箱式审批

19｜Automations 全解：项目级、线程级、跨天续作与定时唤醒
20｜Slack / 移动端 / 收件箱：委派、审批、跟进与归档
21｜主动建议：跨工具拉上下文，生成优先级待办

八、扩展篇——讲透代码之外
> 过关标志：用 Codex 完成一次非编码任务并产出可验收文档

22｜文档与调研：PDF / 表格 / 多源汇总，对接 Notion / Google Docs
23｜浏览器与电脑操作：应用内浏览器标注、GUI 任务、图像原型

九、团队篇——讲透治理与嵌入
> 过关标志：输出个人 Playbook（≤10 条委托模板 + 验收方式）

24｜多角色 Worktree 并行：Dev / Reviewer / On-call 分工
25｜企业治理：`requirements.toml`、托管策略、监控与分析
26｜收官：翻车诊所总复盘、SDK 嵌入、Playbook 导出与综合实战

结束语｜讲透的标准不是「听过」，而是「下次接到任务，知道开哪个入口、下什么委托、跑哪条验收」

## 四、单章体例

每章统一五环节，**宁可少讲一章，也不空泛带过**：

| 环节 | 要求 |
|------|------|
| 机制讲透 | 本章解决的唯一问题，Codex 哪项能力负责 |
| 决策卡 | 何时用 / 何时不用 / 常见误用 |
| 实操演示 | App 或 CLI 完整走一遍，含配置片段与关键命令 |
| 翻车与纠偏 | 至少 1 个典型失败 + 恢复步骤 |
| 过关任务 | 1 个必做练习，产出截图或验收命令输出 |
