# 极客时间专栏立项大纲｜Codex 智能交付实战课

## 一、选题信息

**暂定名称**：Codex 智能交付实战课
**副标题**：从第一张工单到团队级 Agent 工作流
**作者**：袁从德
**交付形态**：极客时间专栏
**预计体量**：开篇词 + 20 讲 + 结束语，可增加少量实战加餐
**配套仓库**：https://github.com/congde/codexDemo.git
**关联图书**：《Codex：AI 驱动的智能编程时代》（邮电出版社）

---

## 二、为什么要做这门课

AI 编程工具已经从代码补全和对话问答，发展到能够读取仓库、修改文件、运行命令、执行测试和参与代码审查的编程智能体。但在实际使用中，开发者仍普遍面临三类问题：

1. **会聊天，不会委托**：能够让 Codex 修改几行代码，却不会把真实需求整理成边界清楚、可以验收的任务。
2. **会生成，不会交付**：Codex 给出了代码，但修改范围、测试结果、风险和交付证据不完整，结果难以进入 PR 和团队流程。
3. **个人偶尔成功，团队无法复用**：缺少项目说明、执行护栏、Review、CI、Skill、Automation 和评测机制，成功经验无法稳定复制。

本专栏不按 Codex 的产品功能逐项讲解，而是让读者沿着一条真实研发交付链路完成训练：

> 接到工单 → 澄清任务 → 委托 Codex → 审查结果 → 运行验收 → 提交 PR → 处理 Review 和 CI → 交接与自动化 → 沉淀团队工作流

课程希望解决的不是“Codex 能不能写代码”，而是：

> **开发者怎样使用 Codex，持续交付一份别人敢接、敢审、敢合的结果。**

## 三、专栏与图书的关系

《Codex：AI 驱动的智能编程时代》和本专栏面向同一主题，但承担不同任务。

### 1. 图书负责建立系统认知

图书是一本正式出版的系统性读物，围绕 AI 编程智能体的认知迁移、上下文工程、工程场景、能力边界和团队治理展开。

它重点回答：

- Codex 为什么不只是代码生成工具；
- AI 编程智能体的能力边界在哪里；
- 上下文、规则、验证和人机分工为什么重要；
- 个人与团队应该如何判断、设计和治理 Agent 工作流。

图书适合系统阅读，帮助读者建立较完整、可迁移的方法论。

### 2. 专栏负责完成实战训练

专栏是一门可以独立学习的实战课。它不沿用图书章节顺序，也不把书中内容改写成短课，而是围绕一条真实研发交付链路组织。

它重点回答：

- 收到一张真实工单后，怎样让 Codex 开始工作；
- 怎样写清目标、上下文、约束和完成标准；
- 怎样审查 Diff，识别越界、漏改和测试凑绿；
- 怎样让结果进入 PR、Review、CI 和交接流程；
- 怎样把反复跑通的流程沉淀为 `AGENTS.md`、Skill、Automation 和 Eval。

专栏强调跟练、失败恢复和交付证据。读者不仅阅读案例，还需要在配套仓库中完成可运行、可检查的任务。

### 3. 二者如何互补

| 对比维度 | 正式出版图书 | 极客时间专栏 |
|---|---|---|
| 核心定位 | 系统方法论与工程认知 | 真实研发交付训练 |
| 主要问题 | 为什么这样设计、边界和判断是什么 | 接到任务后具体怎样做、怎样证明完成 |
| 组织方式 | 按认知与知识体系展开 | 按工单到交付的研发链路展开 |
| 学习方式 | 阅读、理解、迁移 | 跟练、复现、审查、验收 |
| 主要载体 | 论述、案例与方法框架 | Codex 实操、Lab、脚本、Skill 和交付物 |
| 最终收获 | 建立智能编程时代的工程判断 | 形成可直接用于工作的 Codex 交付流程 |

两者会共享少量必要概念，例如上下文、验证和人机分工，但处理方式不同：

- 图书负责解释概念、原理和长期判断；
- 专栏只在任务需要时引入概念，并立即通过真实操作和配套资产验证；
- 专栏不会直接搬用图书章节，也不会要求读者必须先购买或读完图书。

**一句话概括**：

> 图书帮助读者建立判断，专栏帮助读者把判断练成可以交付的工作方式。

## 四、目标读者

### 核心读者

- 已经使用过 ChatGPT、Copilot、Cursor、Codex 等 AI 编程工具，但使用方式仍以聊天和局部生成为主的开发者；
- 希望把 Codex 用于真实代码库、Bug 修复、功能开发、Review 和 CI 的研发人员；
- 希望将个人 AI 编程经验沉淀为团队工作流的技术负责人。

### 前置基础

- 具备基本的软件开发和 Git 使用经验；
- 能够阅读简单代码、测试和终端输出；
- 不要求有复杂 Agent、MCP 或自动化经验。

### 学完后的能力

读者完成课程后，应能够：

1. 将模糊需求整理成 Codex 可以执行和验收的任务；
2. 为 Codex 提供正确上下文，并设置必要的工程边界；
3. 使用 Codex 完成勘察、Bug 修复、功能开发和长任务；
4. 审查 Codex 的修改，并用测试和交付证据判断结果；
5. 让 Codex 参与 PR、Review、CI、并行任务和交接；
6. 将稳定流程沉淀为 Skill、Automation 和团队 Playbook；
7. 使用 Rubric 和 Eval 持续改进 Agent 工作流。

## 五、课程设计原则

### 1. 按交付链路组织，不按功能菜单组织

Codex App、CLI、Cloud、MCP、Skills、Automations 等能力，只在真实任务需要时出现。课程不会单独罗列产品功能。

### 2. 每讲解决一个真实工程问题

每讲从一张工单、一次失败、一个 PR 或一项交付要求开始，让读者理解为什么需要当前方法。

### 3. 解释决策，而不是记录操作流水

正文不仅展示“执行了什么命令”，还会解释：

- 为什么此时要这样做；
- Codex 获得了什么上下文和反馈；
- 如果跳过会产生什么风险；
- 哪些判断和责任不能交给 Agent。

### 4. 重要章节配套可运行资产

根据内容需要提供：

- starter / solution Lab；
- 测试与验收脚本；
- 项目级 `AGENTS.md`；
- Skill 目录、模板和校验脚本；
- PR、Review、CI、handoff 和 Eval 样例。

配套代码不是展示答案，而是用来复现问题、验证方法和承接过关任务。

### 5. 图片承担解释和证据作用

在合适位置安排机制图、Codex 操作截图、Diff、失败与成功对照图。每张图片都有明确图号和建议图片名，避免装饰性配图。

## 六、课程结构

### 开篇词｜不要先学功能，先接一张真实工单

说明课程为什么从交付任务开始，以及“能生成代码”和“能完成交付”的区别。

### 第一篇｜接单与闭环：先独立交付一张工单

读者先完成一次小而完整的研发任务，再理解委托、审查和验收。

**第 1 讲｜别先学提示词：用 Codex 交付第一张热修复工单**
通过一个可复现的排序 Bug，讲清 Codex 如何读取任务、运行测试、修改代码并提交证据。配套 starter、solution 和验收脚本。

**第 2 讲｜接单与计划：把模糊需求变成可执行 Brief**
讲清 Goal、Context、Constraints、Done when，学习在修改前收敛任务边界和开放问题。

**第 3 讲｜审 Diff 与验收：如何接住 Codex 的结果**
从文件范围、行为变化、测试证据和维护成本四个层次审查结果，处理越界、漏改和测试凑绿。

**本篇交付结果**：一份可执行 Brief、一个最小代码 Diff、一组真实验收结果和一次人工放行判断。

### 第二篇｜上下文与护栏：让 Codex 在正确边界内工作

读者将第一次任务中的重复要求和风险沉淀为项目级能力。

**第 4 讲｜`AGENTS.md`：Codex 每次开工前都会读取的工程接口**
解释 Codex 如何发现和组合项目指令，如何从真实失败中生成稳定规则。配套仓库根 `AGENTS.md`。

**第 5 讲｜Rules、Hooks 与 Permissions：让边界真正执行**
把“不要乱改、不要泄露、不要绕过检查”从文字提醒变成规则、拦截和审批。

**第 6 讲｜上下文裁剪与 MCP：让 Codex 看到正确资料**
区分必需、按需和禁止上下文，并通过 MCP 获取可追溯的外部信息，避免缺信息时猜测。

**本篇交付结果**：项目级 Agent 说明、可执行护栏和一份可追溯的任务上下文包。

### 第三篇｜高频研发任务：从能改代码到能稳定改对

读者进入真实代码库，完成不同风险和规模的研发任务。

**第 7 讲｜陌生代码库勘察：先交地图，再交代码**
输出职责地图、调用链、测试覆盖和影响分析，让勘察结果成为下一张工单的输入。

**第 8 讲｜Bug 定位与回归：先红后绿，证明问题真的被修复**
从失败行为定位根因，完成最小修改，并建立可以阻止复发的回归测试。

**第 9 讲｜功能开发：交付一个可审查的跨文件竖切**
从用户行为切分需求，处理跨文件契约、默认行为和异常态。

**第 10 讲｜长任务交付：重构、迁移、Checkpoint 与回滚**
为长时间任务建立行为基线、阶段验收和回滚路径，避免一次修改到底。

**本篇交付结果**：勘察报告、Bug 修复与回归测试、功能竖切，以及一份可回滚的长任务计划。

### 第四篇｜PR、Review 与 CI：让结果能够被团队接受

读者不再以“代码写完”为完成标准，而是让结果进入团队质量流程。

**第 11 讲｜PR 描述与交付证据：让 Reviewer 快速做判断**
将背景、范围、测试、风险和未决事项组织为可复现的 PR 交付接口。

**第 12 讲｜Review 与改评闭环：Finding 要可修、可验、可拒绝**
让 Codex 参与 Review，并由实现方处理意见、Reviewer 重新验证。

**第 13 讲｜CI 失败自修复：自动推进，但不自动放行**
使用非交互任务和 CI 自动分析失败，同时保持分支隔离、权限边界和人工放行。

**第 14 讲｜PR 为什么会被拒：从失败结果反推工作流**
区分代码问题、任务错位、工作流约束和证据不足，学习重写 Brief 或拆分方案。

**本篇交付结果**：完整 PR、Review findings、改评记录、CI 验收和一次拒绝复盘。

### 第五篇｜复用、并行与自动化：让稳定流程持续推进

读者将已经跑通的个人流程转化为可接手、可复用和可自动运行的工作方式。

**第 15 讲｜Worktree、Cloud 与 Handoff：并行任务如何不串线**
隔离文件、分支和上下文，并让下一位成员无需阅读原线程即可继续任务。

**第 16 讲｜Skills：把“会做一次”变成可复用工作流**
从重复任务中提炼适用场景、标准步骤、模板、脚本和验收。配套真实 `repo-readiness` Skill。

**第 17 讲｜Automations：只自动化已经稳定的工作流**
设计触发条件、权限、审批、失败降级和收件箱，避免自动化放大含糊流程。

**本篇交付结果**：一份可接手 Handoff、一个经过验证的 Skill，以及一条低风险 Automation 设计。

### 第六篇｜扩展交付与持续改进：从个人熟练到团队能力

读者把 Codex 用于代码之外的交付，并建立持续评估和团队推广机制。

**第 18 讲｜非代码交付：文档、数据与浏览器证据**
使用 Codex 处理调研、数据和浏览器验收，区分事实、推断、建议和未知。

**第 19 讲｜Traces、Rubric 与 Evals：怎样证明工作流稳定有效**
保存执行轨迹、设计评分标准，用多个样本比较工作流修改前后的表现。

**第 20 讲｜毕业综合交付与团队 Playbook**
完成一条从工单、实现、PR、Review、CI 到 Handoff 和复盘的完整链路，并导出团队可复用工作流。

**本篇交付结果**：带证据的非代码产物、最小 Eval、端到端毕业交付和个人/团队 Playbook。

### 结束语｜真正的过关，是下一张工单仍然知道怎样交付

## 七、单讲内容形态

每讲预计包含以下内容：

| 内容 | 作用 |
|---|---|
| 真实问题或工单 | 建立任务背景和交付压力 |
| Codex 机制解释 | 讲清 Codex 如何读取、执行、反馈和受限 |
| 完整实操案例 | 展示从输入到结果的关键过程 |
| 人工判断点 | 明确不能交给 Agent 的责任 |
| 翻车与恢复 | 展示高概率失败和恢复方法 |
| 图片与配套代码 | 解释机制、展示证据、支持复现 |
| 过关任务 | 让读者独立产出可检查交付物 |

正文避免写成命令流水账。操作过程必须服务于任务判断和交付结果。

## 八、配套仓库与交付物

配套 Git 仓库 `codexDemo` 是本专栏的重要组成部分，不是普通示例代码下载包。

仓库将逐步提供：

- 与章节对应的可运行 Lab；
- 可以复现失败的 starter 和通过同一合同的 solution；
- 项目级 `AGENTS.md`；
- Skill、模板和确定性校验脚本；
- PR、Review、CI、Handoff 和 Eval 样例；
- 统一的 `make check` 验收入口。

每项配套资产都需要能够实际运行。正文提到的文件、命令和验收必须与仓库保持一致。

## 九、差异化价值

与常见 AI 编程课程相比，本专栏的差异主要体现在：

1. **不以提示词和功能介绍为主线**，而以真实研发交付链路为主线；
2. **不仅讲 Codex 怎样生成代码**，还重点讲怎样审查、验证、拒绝和恢复；
3. **不仅覆盖个人编码任务**，还覆盖 PR、Review、CI、交接、Skill、Automation 和 Eval；
4. **每个重要方法都有配套资产和验收证据**，读者可以复现，而不是只看演示；
5. **明确人机责任边界**，强调最终目标、风险取舍和放行责任仍由人承担。

---

## 十、预期成果

专栏完成后，读者不仅会使用 Codex，还会拥有一套能够迁移到自己项目中的交付资产：

- 一套可复用的任务 Brief 写法；
- 一份适合自己项目的 `AGENTS.md`；
- 一组审 Diff、Review 和验收方法；
- 一个经过验证的 Skill；
- 一条低风险 Automation 设计；
- 一套用于持续改进的 Rubric 和 Eval；
- 一份个人或团队 Codex Playbook。

最终希望读者形成的不是“让 AI 多写一点代码”的技巧，而是：

> **把 Codex 放进一条可解释、可验证、可审查、可接手、可持续改进的研发交付流程。**
