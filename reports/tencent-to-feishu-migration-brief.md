# 委托 Brief：8 人项目团队是否值得从腾讯文档迁移到飞书文档

## Codex 执行前评审

### 评审结论

**状态：足以执行桌面调研与试点设计，但不足以直接作出最终迁移决定。**

这份 Brief 已明确要支持的决定、团队规模、比较范围、证据要求、交付物和验收条件。Codex 可以在不依赖额外口头说明的情况下开始调研，并给出“继续使用、停止评估或进入试点”的建议。

最终迁移决定仍需要团队负责人放行，因为当前缺少团队真实使用数据，以及代表性文档迁移后的格式、评论、权限和历史记录保真度证据。这些内容已被保留为 Open questions，不应由 Codex 自行假设。

### 执行风险与处理

| 风险 | 是否阻塞调研 | 处理方式 |
|---|---|---|
| 团队痛点尚未量化 | 否 | 作为 Unknown，提供访谈问题和采集方法 |
| 当前版本、价格与权益可能变化 | 否 | 只引用执行时可访问的官方来源，并记录访问日期 |
| 迁移保真度无法仅靠公开资料确认 | 否 | 不写成事实，设计代表性迁移测试 |
| 最终决策权归属不清 | 否 | 明确由团队负责人作出最终迁移决定 |

### 开工指令

Codex 应先复述它理解的决策目标、范围边界和关键未知，再开始检索。若无法访问官方来源，应停止事实性结论的撰写，记录阻塞项，不得用搜索摘要、论坛或经验猜测替代。

## Goal

为一个长期使用腾讯文档的 8 人项目团队产出一份可复核的迁移评估，判断团队是否值得进入“飞书文档两周试点”，并为团队负责人在以下三个选项中作出决定提供依据：

1. 继续使用腾讯文档；
2. 开展有限范围的飞书文档试点；
3. 暂停评估，先补齐关键未知。

本任务不直接批准采购、全量迁移或公司级办公平台替换。

## Context

- 使用者是一个 8 人项目团队，目前持续使用腾讯文档。
- 典型资料包括项目方案、会议纪要、任务或进度表、复盘记录。
- 评估重点不是单纯比较功能数量，而是判断切换后能否改善真实协作流程，并且收益是否足以覆盖学习、迁移和双平台并行成本。
- 需要重点关注：文档协作，会议、日历与任务联动，知识沉淀，多维表格或项目跟踪，外部协作，权限管理，迁移保真度和价格。
- 当前尚未提供团队的文档数量、付费版本、外部协作者比例、合规要求和具体痛点数据。
- 报告读者是团队负责人；最终迁移决定由团队负责人作出。

## Constraints

- 仅评估 8 人项目团队的文档与项目协作场景，不扩大到公司级即时通信、HR、审批或整体办公平台替换。
- 不因飞书功能更多就默认值得迁移；结论必须结合团队需求、采用成本和迁移风险。
- 不只比较价格，必须覆盖协作流程、外部协作、权限、格式、评论、历史记录和迁移成本。
- 产品功能、版本权益和价格只能依据执行时可访问的官方产品页、帮助中心或价格页；每条事实必须附 URL 和访问日期。
- 搜索结果摘要、媒体文章、厂商宣传口号和个人经验不得直接写入 Facts。
- 必须区分 `Facts`、`Inferences`、`Recommendations` 和 `Unknowns`，不得把推断写成事实。
- 无法验证的团队需求、价格、迁移保真度或功能限制必须标记为 Unknown，不得自行补全。
- 不执行真实账号配置、付费购买、数据迁移或删除操作。
- 不推荐全量迁移，除非后续代表性试点达到预先定义的通过标准。
- 正文不超过 2,500 个中文字符，不含来源清单、评分表与附录。

## Done when

- 输出保存为 `reports/tencent-to-feishu-migration-assessment.md`。
- 报告包含 `Executive summary`、`Facts`、`Inferences`、`Recommendations`、`Unknowns`、`Pilot design` 和 `Sources`。
- `Executive summary` 明确建议三个选项之一：继续使用、进入两周试点或暂停评估补证，并说明主要依据。
- 至少核验 6 条对决策有影响的关键事实，其中至少包括 1 条价格或版本权益事实、1 条外部协作事实、1 条权限事实和 1 条迁移或导入导出事实。
- 每条 Fact 使用 `F1`、`F2` 等编号，附官方来源 URL、访问日期和来源是否充分支持该事实的人工复核结果。
- 每条 Inference 使用 `I1`、`I2` 等编号，并声明 `Supports: F...`。
- 每条 Recommendation 使用 `R1`、`R2` 等编号，并声明 `Supports: I...`。
- 每条 Unknown 使用 `U1`、`U2` 等编号，并包含 `Impact:` 和 `Next check:`。
- 提供一张加权评分表，至少覆盖协作收益、迁移风险、采用成本、外部协作、权限与安全、总拥有成本；权重和评分依据必须可见。
- 提供一个不超过两周、只覆盖一个代表性项目空间的试点方案，写明样本、参与者、成功指标、停止条件和回滚方式。
- 试点成功指标至少包括：资料查找耗时、任务遗漏情况、团队采用率、外部协作体验，以及格式、评论、权限和历史记录保真度。
- 报告必须明确指出哪些结论可由公开证据支持，哪些只能通过团队访谈或迁移试点验证。
- 完成后，Codex 对照本 Brief 做一次自评审，逐项报告 Done when 的通过、失败或未执行状态，不得只声明“已完成”。

## Open questions

- 团队当前使用腾讯文档的具体版本与实际费用是多少？
- 团队最希望解决的三个协作痛点是什么，发生频率和影响有多大？
- 团队是否愿意把主要沟通、会议、日历或任务管理一起迁入飞书？
- 外部客户、供应商或临时协作者占比多高，他们能否顺利使用飞书？
- 团队是否有数据存储、账号管理、审计或其他合规要求？
- 现有项目空间包含多少文档、表格和附件，是否依赖复杂格式、评论、权限与历史版本？
- 代表性迁移测试能否保留格式、评论、权限、链接、附件和历史记录？
- 团队可接受的试点时间、学习成本、双平台并行期和付费预算是多少？
